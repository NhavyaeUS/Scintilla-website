SECTION 01  EXECUTIVE OVERVIEW 
1. Executive Overview 1.1 Project Purpose Scintilla is a student-run literary and arts magazine that collects creative submissions through a two-page web interface. The frontend — index.html (landing) and submit.html (submission form) — is already fully built and must not be modified. This PRD defines the complete backend and integration layer that will power the submission pipeline, file storage, and data tracking system.  1.2 Scope Statement This document covers backend architecture ONLY. No frontend changes are included or implied. The backend must seamlessly attach to the existing HTML forms without requiring any modifications to them.  1.3 System Goals • Accept and validate multi-file, multi-category form submissions from submit.html ◦ Upload submitted files to a designated Google Drive account, organised into category subfolders Log all submission metadata into a Google Sheet for editorial review Provide secure, validated API endpoints with proper error handling Support concurrent submissions without data loss or race conditions Be deployable to a cloud host with HTTPS, environment-variable secrets, and structured logging  1.4 Out of Scope • Any modification to index.html or submit.html ◦ A user authentication/login system (submissions are anonymous or lightly identified) A CMS, editorial review UI, or admin dashboard (managed directly in Google Sheets) Email confirmation to submitters (optional future enhancement, not in v1)

SECTION 07  FAILURE HANDLING & RETRY LOGIC 
7. Failure Handling & Retry Logic 7.1 Retry Strategy Google API calls (Drive upload, Sheets append) are wrapped in an exponential back-off retry helper. Transient failures (5xx from Google, network timeouts) are retried up to 3 times before the request is considered failed. // src/utils/retry.js async function withRetry(fn, maxAttempts = 3) {   for (let attempt = 1; attempt <= maxAttempts; attempt++) {     try {       return await fn();     } catch (err) {       if (attempt === maxAttempts) throw err;       const delay = Math.pow(2, attempt) * 500; // 1s, 2s, 4s       logger.warn("Retrying API call", { attempt, delay, error: err.message });       await new Promise(r => setTimeout(r, delay));     }   } }  7.2 Partial Failure Handling If a multi-entry submission (cart with multiple works) partially succeeds — for example, two uploads succeed and one fails — the backend: • Returns HTTP 207 Multi-Status with a per-entry status array ◦ Logs the partial failure event with all successfully processed entry data Does NOT roll back already-completed Drive uploads or Sheet rows (append-only system) Includes the failed entry's error message in the response so the browser can display it
